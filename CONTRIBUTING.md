# Contributing to gitoma-bench-blast

Welcome! We welcome contributions from everyone. If you'd like to help improve this project, here is a guide on how to contribute.

## How to Contribute

Contributions can take many forms, such as reporting bugs, suggesting features, or submitting code fixes.

### 🐛 Bug Reports

If you find a bug, please follow these steps:

1.  **Reproduce the issue:** Try to reproduce the bug consistently.
2.  **Describe the steps:** Provide clear, step-by-step instructions on how to trigger the bug.
3.  **Provide context:** Include relevant logs, screenshots, or configuration details that help diagnose the problem.
4.  **Report:** Open a new issue on the repository with all this information.

### ✨ Feature Requests

If you have an idea for a new feature:

1.  **Describe the goal:** Clearly explain what you want to add.
2.  **Explain the benefit:** Describe why this feature would be useful for the project users.
3.  **Propose implementation:** If possible, suggest how it might be technically implemented.

### 💻 Code Contributions

If you are ready to submit code changes:

1.  **Fork the repository:** Start by forking the project to your account.
2.  **Clone and branch:** Clone your fork and create a new branch for your changes (e.g., `git checkout -b feature/my-fix`).
3.  **Make changes:** Implement your fix or feature.
4.  **Test thoroughly:** Ensure your changes pass all existing tests and that you have added any necessary tests for your new code.
5.  **Commit:** Commit your changes with a clear, descriptive commit message (following [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/)).
6.  **Pull Request:** Push your branch and open a Pull Request against the main branch of `gitoma-bench-blast`.

## Development Workflow

We use a standard workflow:

*   **Testing:** All contributions must include tests that cover the changes.
*   **Style/Linting:** Please adhere to the project's established coding style guides (if any).

Thank you for your contribution!
