# Contributing to gitoma-bench-blast

Welcome! We welcome contributions from everyone. If you'd like to help improve this project, here is a guide on how to contribute.

## How to Contribute

Contributions can take many forms, such as reporting bugs, suggesting features, or submitting code fixes.

### 🐛 Bug Reports
If you encounter a bug, please follow these steps:

1.  **Reproduce the issue:** Try to reproduce the bug consistently.
2.  **Describe the steps:** Provide clear, numbered steps to reproduce the issue.
3.  **Provide context:** Include relevant logs, screenshots, or configuration details that might help diagnose the problem.
4.  **Suggest a fix (optional):** If you have an idea for how to fix it, describe your thoughts.

### ✨ Feature Requests
If you have an idea for a new feature:

1.  **Describe the goal:** Clearly explain what you want to add.
2.  **Explain the benefit:** Describe why this feature would be useful for the project's goals.
3.  **Provide usage examples:** Show how this feature would be used.

### 💻 Code Contributions
If you are looking to contribute code:

1.  **Fork the repository:** Start by forking the project to your account.
2.  **Clone and branch:** Clone your fork and create a new branch for your changes (e.g., `git checkout -b feature/my-new-thing`).
3.  **Make your changes:** Implement the requested feature or fix.
4.  **Test thoroughly:** Ensure your changes pass all existing tests and any new tests you introduce.
5.  **Commit thoughtfully:** Commit your changes with clear, descriptive commit messages (following [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/)).
6.  **Pull Request:** Push your branch and open a Pull Request against the main repository.

## Development Setup

To contribute effectively, please ensure you have the project set up locally:

1.  Clone the repository.
2.  Install dependencies (check `pyproject.toml` for installation instructions).
3.  Run tests (`pytest` or equivalent) to ensure your changes haven't introduced regressions.

## Commit Message Conventions

We use [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) to keep our commit history clean and useful for generating changelogs.

*   **feat:** A new feature has been introduced.
*   **fix:** A bug has been fixed.
*   **docs:** Documentation only changes.
*   **chore:** Routine maintenance tasks (e.g., updating build scripts).
*   **refactor:** A code change that neither fixes a bug nor adds a feature.

Thank you for your contribution!
